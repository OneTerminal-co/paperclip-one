FinOps Agent — cada 8 horas
El contador que nunca duerme y que ningun dev quiere ser
Problema real: el 70% de las PYMEs cloud no sabe exactamente en qué se va su dinero. La factura llega a fin de mes y nadie la entiende. Este agente convierte esa caja negra en visibilidad diaria accionable.
Que hace exactamente
  • Consulta Cost Explorer org-wide y desglosa por cuenta, servicio, tag y equipo
  • Detecta anomalias: gasto que supera el umbral historico en mas del 20%
  • Identifica recursos idle: EC2 con CPU menor al 5%, RDS sin conexiones, EBS sin montar
  • Calcula rightsizing: compara familia de instancia actual vs recomendacion de Compute Optimizer
  • Revisa Reserved Instances y Savings Plans: detecta RI sin usar o sin renovar
  • En Azure: analiza suscripciones via Cost Management + Advisor recommendations
  • Genera reporte ejecutivo semanal: "esta semana gastaste $X, detecte $Y de ahorro posible"
Escenarios concretos que resuelve
Escenario 1 — Ambiente dev siempre encendido
El agente detecta que 8 instancias EC2 t3.large en la cuenta dev llevan 720h encendidas este mes sin actividad significativa.
Output: "Costo innecesario estimado: $340 USD/mes. Recomendacion: schedule de apagado automatico entre 10pm y 6am."
Escenario 2 — Anomalia de transferencia de datos
Gasto en data transfer sube 300% en 48 horas. El agente cruza con CloudTrail y detecta un job de ETL mal configurado que replica datos entre regiones innecesariamente.
Output: alerta inmediata al CEO Agent con causa probable y ticket generado.
Escenario 3 — RI expirando sin renovar
Detecta 3 Reserved Instances de RDS que expiran en 14 dias. Sin renovacion, el costo se triplica al volver a on-demand.
Output: reporte al CTO con costo comparativo y recomendacion de Savings Plan alternativo.
Opciones adicionales a considerar
Agente FinOps con modo write (apagado automatico)
En lugar de solo recomendar, el agente ejecuta: apaga instancias dev fuera de horario laboral, elimina snapshots EBS huerfanos, borra AMIs sin usar. Requiere permisos write aprobados por el board.
Alto valor — ahorro real automatizado, no solo reportes
Forecasting mensual con alertas de presupuesto
El agente proyecta el gasto del mes en curso basado en los primeros 7 dias. Si la proyeccion supera el budget aprobado, alerta al CTO antes de que ocurra, no despues.
Alto valor — especialmente en empresas con control estricto de presupuesto
Tag enforcement y chargeback interno
Detecta recursos sin tags obligatorios (proyecto, equipo, ambiente). Genera reporte de chargeback por equipo. Muy util en empresas con multiples equipos compartiendo una cuenta.
Valor medio — requiere que el cliente tenga cultura de tagging previa
Metrica de valor para el pitch
El benchmark de mercado dice que el 30-35% del gasto cloud es desperdicio en empresas sin FinOps formal. En un cliente que paga $10.000 USD/mes en AWS, el agente puede identificar $3.000 USD/mes de ahorro — 6x el costo del plan Professional de OpsFleet.
Security Agent — cada 12 horas
El que encuentra los huecos antes de que los encuentre otro
Problema real: la mayoria de brechas de seguridad en PYMEs cloud no son ataques sofisticados — son configuraciones mal hechas que llevan meses sin que nadie las vea. Un bucket S3 publico, un security group con puerto 22 abierto al mundo, credenciales de acceso de un ex-empleado que nunca se revocaron.
Que hace exactamente
  • Revisa SecurityHub org-wide: agrega findings de todas las cuentas y prioriza por severidad CRITICO / ALTO / MEDIO
  • Audita IAM: usuarios con MFA desactivado, roles con permisos excesivos (AdministratorAccess en produccion), access keys sin rotar en mas de 90 dias
  • Revisa Security Groups: reglas 0.0.0.0/0 en puertos criticos (22, 3389, 5432, 3306)
  • Detecta buckets S3 publicos o con ACLs permisivas no intencionales
  • Revisa GuardDuty: amenazas detectadas, IPs maliciosas, comportamiento anomalo de credenciales
  • En Azure: Defender for Cloud score, recomendaciones por suscripcion, alertas de Sentinel
  • Cruza hallazgos con aws-knowledge-mcp-server para obtener SOPs de remediacion actualizados
Escenarios concretos que resuelve
Escenario 1 — Credencial de ex-empleado activa
El agente detecta un IAM user "jperez" con access key activa, ultimo uso hace 3 dias, pero ese usuario fue dado de baja del sistema de RRHH hace 6 semanas.
Output: alerta CRITICA al CEO Agent. Ticket generado: "Revocar acceso de jperez inmediatamente".
Escenario 2 — Puerto de base de datos expuesto
Security Group de RDS en produccion tiene puerto 5432 abierto a 0.0.0.0/0. Lleva 4 meses asi desde que un dev lo abrio "temporalmente" para hacer debug.
Output: finding CRITICO con instrucciones de remediacion especificas y numero de CVEs que ese vector tiene.
Escenario 3 — GuardDuty: comportamiento anomalo
Un Lambda function empieza a hacer llamadas a IPs en rangos conocidos de C&C malware a las 3am. GuardDuty lo detecta, el Security Agent lo procesa y lo escala con contexto.
Output: alerta inmediata con IP, funcion afectada, patron de trafico y SOP de contencion.
Opciones adicionales a considerar
Rotacion automatica de secrets en Key Vault / Secrets Manager
El agente detecta secrets con mas de 90 dias sin rotar y puede disparar el proceso de rotacion automatica si tiene permisos write. Util para tokens de integraciones, API keys, certificados SSL.
Critico en banca y salud — es un requisito de cumplimiento, no solo buena practica
Escaneo de imagenes de contenedor (SAST/SCA)
Integrar Trivy o Inspector v2 de AWS para escanear imagenes en ECR/ACR en busca de CVEs. El agente reporta vulnerabilidades criticas por imagen y tag.
Alto valor — especialmente si el cliente tiene pipelines de contenedores en produccion
Simulacion de escalada de privilegios (read-only)
El agente analiza las politicas IAM existentes y detecta cadenas de permisos que podrian permitir escalada de privilegios aunque ninguna politica individual sea obvia. Tecnica de IAM privilege analysis.
Valor alto, complejidad alta — diferenciador fuerte para clientes regulados
Limite critico a comunicar al cliente
Este agente detecta y alerta. NO hace penetration testing activo. NO explota vulnerabilidades. Si el cliente necesita un pentest formal, eso es un servicio separado. El agente es postura defensiva continua, no ofensiva.
Infra Ops Agent — cada 4 horas
El SRE que revisa la salud del sistema mientras el equipo duerme
Problema real: los equipos pequenos no tienen on-call formal. Cuando un nodo de Kubernetes falla a las 2am o un RDS se queda sin espacio en disco, se enteran cuando el cliente llama. Este agente corre cada 4 horas y reporta el estado real de la infraestructura.
Que hace exactamente
  • Revisa estado de EC2/VMs: instancias en estado degradado, alta utilizacion de CPU/memoria sostenida
  • Kubernetes/AKS/EKS: nodos NotReady, pods en CrashLoopBackOff, deployments con replicas por debajo del minimo
  • RDS/Azure SQL/Cosmos: espacio en disco, conexiones activas vs max, replication lag en replicas
  • Lambda/Functions: tasa de errores, timeouts, throttling sostenido
  • Load Balancers: healthy host count, 5XX rate en ALB/API Gateway
  • Revisa CDK stack drift: detecta recursos que divergen del estado declarado en IaC
  • Compara configuracion actual vs best practices via cdk-mcp-server y aws-knowledge
Escenarios concretos que resuelve
Escenario 1 — RDS al 90% de capacidad de disco
El agente detecta que la instancia de PostgreSQL en produccion tiene el disco al 91% de capacidad. Al ritmo actual de crecimiento, se llena en 6 dias.
Output: alerta con proyeccion de llenado, opciones de expansion (modify storage) y costo estimado de cada opcion.
Escenario 2 — Pod en CrashLoopBackOff silencioso
Un microservicio de notificaciones lleva 3 horas en CrashLoopBackOff en el namespace de produccion. El equipo no lo sabia porque no habia alerta configurada para ese namespace.
Output: alerta con nombre del pod, logs del ultimo crash, y runbook recomendado del knowledge server.
Escenario 3 — IaC drift detectado
Un dev modifico manualmente un Security Group en la consola de AWS sin actualizar el CDK. El agente detecta que el estado real diverge del declarado en el stack.
Output: reporte de drift con diff exacto entre estado actual y estado esperado por IaC.
Opciones adicionales a considerar
Agente con observabilidad OTel (Prometheus + Grafana)
Si el cliente tiene OpenTelemetry desplegado, el agente puede consultar metricas de aplicacion reales (latencia p99, tasa de errores de negocio) en lugar de solo metricas de infraestructura. Mucho mas valor pero requiere que el cliente tenga OTel instrumentado.
Alto valor — diferencia entre monitoreo de infra y monitoreo de negocio
Agente de DRP (Disaster Recovery)
Verifica periodicamente que los backups se estan ejecutando y son restaurables. Comprueba que los snapshots de RDS son recientes. Valida que el failover de multi-AZ esta configurado. Genera reporte mensual de estado de DRP.
Critico para banca y salud — es lo primero que pregunta una auditoria
Agente de performance (rightsizing proactivo)
Analiza metricas de utilizacion de los ultimos 30 dias y genera recomendaciones de rightsizing: que instancias escalar, cuales reducir, cuales migrar a Graviton. Cruza con el FinOps Agent para calcular impacto economico.
Valor medio-alto — genera ahorro y mejora de performance al mismo tiempo
Knowledge Agent — on-demand via tickets
El arquitecto de referencia disponible 24/7 para preguntas tecnicas
Problema real: el equipo tecnico de la PYME pierde horas buscando en documentacion AWS/Azure, Stack Overflow y foros. Muchas decisiones de arquitectura se toman con informacion desactualizada o incompleta porque buscar bien es caro en tiempo.
Que hace exactamente
  • Responde preguntas tecnicas especificas sobre AWS y Azure via aws-knowledge-mcp-server (docs en tiempo real)
  • Busca y resume best practices para un servicio especifico: "como configurar VPC endpoints para RDS"
  • Genera runbooks: "pasos para hacer un failover de RDS Multi-AZ de forma segura"
  • Evalua decisiones de arquitectura: "es mejor usar SQS o EventBridge para este caso de uso"
  • Documenta lo que el equipo hace: convierte conversaciones en documentacion estructurada
  • Responde sobre disponibilidad regional de servicios: "esta disponible X en sa-east-1"
Escenarios concretos que resuelve
Escenario 1 — Decision de arquitectura urgente
El equipo necesita decidir entre API Gateway HTTP API vs REST API para un nuevo microservicio. El Knowledge Agent consulta la documentacion actual, compara features, limitaciones y precios, y genera un ADR (Architecture Decision Record).
Output: ADR completo con pro/contras, precio estimado y recomendacion justificada.
Escenario 2 — Runbook para un incidente
El Infra Ops Agent detecta que un cluster EKS tiene nodos en estado NotReady. El Knowledge Agent genera el runbook de diagnostico y recuperacion especifico para EKS 1.29 en la region del cliente.
Output: runbook paso a paso con comandos kubectl exactos y criterios de exito.
Escenario 3 — Evaluacion de nueva feature de AWS
AWS anuncia una nueva feature de RDS. El equipo quiere saber si aplica a su caso. El Knowledge Agent busca la documentacion, extrae los requisitos y evalua si el cliente puede adoptarla.
Output: resumen ejecutivo de la feature + evaluacion de applicabilidad al entorno del cliente.
Limitacion importante a ser honesto con el cliente
Este agente es tan bueno como el contexto que tiene del entorno del cliente. Sin un SKILLS.md bien construido que describa la arquitectura, los servicios usados y las decisiones ya tomadas, las respuestas son genericas. El valor real emerge despues de 2-4 semanas de uso cuando el agente ha acumulado contexto especifico.
CI/CD Agent — cada 6 horas + webhooks
El que detecta que el pipeline esta roto antes de que el dev lo intente usar
Problema real: los pipelines de CI/CD se rompen silenciosamente. Un job de noche falla, nadie lo ve hasta la manana siguiente cuando alguien quiere hacer deploy. O peor: el pipeline pasa pero el deploy va a produccion con una vulnerabilidad porque el step de SAST estaba deshabilitado "temporalmente".
Que hace exactamente
  • Monitorea GitHub Actions / Azure DevOps pipelines: workflows fallidos, jobs colgados, runners sin capacidad
  • Detecta pipelines que pasaron pero con steps de seguridad deshabilitados o skipped
  • Revisa tiempo de ciclo (cycle time) y detecta degradacion: si el pipeline que tardaba 8 minutos ahora tarda 25, investiga por que
  • Analiza tasa de fallo por rama, por servicio, por tipo de test — encuentra los flaky tests
  • Revisa que los ambientes de staging se estan actualizando correctamente despues de cada merge
  • Detecta deploys a produccion fuera de ventana de cambio aprobada
  • Genera DORA metrics: deployment frequency, lead time, MTTR, change failure rate
Escenarios concretos que resuelve
Escenario 1 — Flaky test que bloquea el equipo
El agente detecta que el test "integration/payment-flow" falla el 40% de las veces sin cambios en el codigo. El equipo ha estado reejecutando el pipeline manualmente 2 veces por dia.
Output: reporte con el test especifico, patron de fallos (falla mas en horario de carga) y recomendacion de aislamiento.
Escenario 2 — Deploy a produccion sin aprobacion
Un dev hace merge directamente a main sin pasar por el proceso de aprobacion del branch protection. El pipeline corre y despliega a produccion automaticamente.
Output: alerta inmediata al CEO Agent con el commit, el autor y el timestamp. Genera ticket de revision.
Escenario 3 — DORA metrics deterioradas
El deployment frequency bajo de 8 deploys/semana a 2 en el ultimo mes. El agente cruza con los datos de pipeline y detecta que el culpable es un step de pruebas de integracion que paso de 5 a 22 minutos.
Output: reporte de DORA con tendencia, causa probable y recomendacion de optimizacion.
Por que este agente es un diferenciador fuerte de OpsFleet
Ningun otro servicio de monitoreo cloud incluye CI/CD como parte del paquete base. Es el puente entre operaciones e ingenieria. El CTO puede ver en un dashboard: "el equipo despliega 5 veces por semana con 98% de exito" — eso es valor directo de negocio medible.
Compliance Agent — semanal + pre-auditoria
El que prepara la evidencia antes de que llegue el auditor
Problema real: una auditoria ISO 27001, SOC2 o de la SFC tarda semanas en preparar la evidencia manualmente. El equipo tecnico dedica 2-3 semanas completas a recolectar logs, configuraciones y reportes. Este agente hace eso de forma continua y entrega la evidencia lista.
Que hace exactamente
  • Evalua controles contra frameworks: CIS AWS Benchmark, NIST CSF, ISO 27001 Annex A via awslabs OSCAL MCP server
  • Genera evidencia de cumplimiento automatica: "el cifrado en reposo esta activo en todos los servicios de datos desde fecha X"
  • Detecta desviaciones de la linea base de compliance antes de que sean hallazgos de auditoria
  • Rastrea que los controles compensatorios documentados siguen activos
  • Genera reporte de estado de compliance por control, por servicio y por suscripcion/cuenta
  • En Azure: evalua Azure Policy compliance report y genera evidencia de remediacion
  • Mantiene un historial de estado de compliance para demostrar mejora continua
Escenarios concretos que resuelve
Escenario 1 — Preparacion de auditoria SOC2
El cliente tiene auditoria SOC2 Type II en 6 semanas. El agente genera el reporte de estado de los controles CC6 (acceso logico) y CC7 (operaciones del sistema) con evidencia lista para el auditor.
Output: reporte de 47 controles con estado PASS/FAIL/N/A y evidencia exportable en PDF.
Escenario 2 — Control de cifrado caido silenciosamente
Alguien creo un bucket S3 nuevo para un proyecto sin activar el cifrado por defecto. El control CIS 2.1.1 falla. El agente lo detecta en 6 horas.
Output: finding con control especifico violado, recurso afectado, y pasos de remediacion.
Escenario 3 — Reporte SFC para fintech
Cliente fintech regulado por la SFC necesita demostrar que los datos de clientes estan cifrados y el acceso esta auditado. El agente genera el reporte mensual de accesos a datos sensibles via CloudTrail.
Output: reporte de accesos exportable, firma temporal de integridad del log, listo para presentar al regulador.
El segmento que mas paga por este agente
Fintech reguladas por SFC, clinicas y EPS con datos sensibles, y empresas que quieren ISO 27001 para poder vender a enterprise. Para ellas el compliance no es nice-to-have — es bloqueador de negocio. Un agente que automatiza la evidencia puede justificar solo el plan Enterprise completo.
CEO Agent — diario
El coordinador que convierte ruido tecnico en decisiones ejecutivas
Problema real: el CTO recibe demasiadas alertas de demasiados sistemas y no puede distinguir lo critico de lo importante de lo irrelevante. El CEO Agent filtra, prioriza, agrupa y presenta una sola vista ejecutiva de lo que paso, lo que importa y lo que requiere decision.
Que hace exactamente
  • Recibe outputs de todos los agentes (FinOps, Security, Infra, CI/CD, Compliance)
  • Prioriza hallazgos: CRITICO (requiere accion hoy), IMPORTANTE (esta semana), INFORMATIVO (reporte mensual)
  • Genera el reporte ejecutivo diario: 1 pagina, sin jerga tecnica, orientado a decision
  • Escala al board (CTO) solo lo que requiere aprobacion humana — no satura con ruido
  • Coordina agentes cuando un hallazgo requiere accion cruzada (ej: Security detecta anomalia, Infra verifica el recurso, Knowledge genera el runbook)
  • Mantiene el contexto del entorno del cliente en SKILLS.md actualizado
  • Genera el reporte semanal de salud del sistema listo para presentar en el standup del CTO
Escenarios de coordinacion
Coordinacion multi-agente
Security Agent detecta una anomalia de credenciales. CEO Agent activa: (1) Infra Agent para verificar que recursos accedio esa credencial, (2) Knowledge Agent para obtener el SOP de contencion, (3) alerta al board con resumen de los 3 inputs.
El CTO recibe un solo mensaje con contexto completo, no 3 alertas separadas que tiene que correlacionar el.
Reporte ejecutivo semanal
Todos los viernes a las 8am, el CEO Agent sintetiza la semana: gasto cloud (+X% vs semana anterior), hallazgos de seguridad (N criticos, M resueltos), salud de infra (uptime 99.8%), estado de pipelines (Y deploys, Z% exito).
Un solo email/Slack al CTO. Sin abrir 5 dashboards distintos.
Por que el CEO Agent es el producto en si mismo

Los otros agentes son la tecnologia. El CEO Agent es la interfaz. Lo que el cliente compra no es "acceso a awslabs/mcp" — compra recibir cada manana un reporte claro de que esta pasando con su infraestructura sin tener que entender CloudWatch, SecurityHub ni Cost Explorer. El CEO Agent es la capa de traduccion entre la complejidad tecnica y la decision ejecutiva.

El argumento de venta del CEO Agent
"Tu CTO recibe un solo mensaje cada manana con todo lo que importa. No 47 alertas de CloudWatch, no 12 emails de SecurityHub, no el dashboard de Cost Explorer que nadie abre. Un mensaje. Una decision. Eso es OpsFleet."
